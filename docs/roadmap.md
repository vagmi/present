## What we're building

**Present** is a Canva-like presentation tool: a Konva.js canvas editor for
building slides, AI image generation via **nano banana** (Google's Gemini image
model) through the **fal** API, shipping with standard templates.
· **Users:** teams/organizations (every record scoped to a Clerk `orgId`, with
roles/memberships)
· **Core resources:** presentations (decks) · slides · templates · assets
(uploaded images) · image generations (AI assets)
**First journey:** sign in → create a deck (blank **or** from a standard
template) → edit slides on the Konva canvas → see them saved.

**Deferred by choice:** AI image generation lands in Phase 5; subscription
billing (the `billing-polar` skill) in Phase 6.

## Conventions every phase respects (from AGENTS.md)

- Strict layering: **controller → service → repository → Drizzle/D1**; never skip
  a layer. Controllers never touch the DB; services never import Hono.
- **Every repository query is scoped by `org_id`**, derived from the Clerk
  session (`requireOrg`), never trusted from the client.
- Authorization rules live in `app/lib/capabilities.ts` and are enforced with
  `requireCapability(can.x)`, gating on the **Clerk session role**
  (`c.var.orgRole`), never the `memberships.role` mirror.
- Validate external input with zod at the controller edge
  (`workers/api/validation.ts`).
- Every layer ships a test; `tests/` mirrors `workers/api/`.
- Side effects (fal calls, webhooks, email) run in `waitUntil(...)`.
- The **copy-`items` recipe** in `docs/workshop.md §2` is the concrete procedure
  for each new resource — reference it, don't reinvent it.

## Phases

### [x] Phase 1 — Presentations (decks)

**Goal:** a user can sign in and create, list, rename, and delete presentations
(title + metadata only — no slides yet). This is the smallest end-to-end slice
and reuses the working `items` code.

**Slice:** schema `presentations` → `presentations-repo` → `presentations-service`
→ `presentations-controller` → route `app/routes/app/presentations-list.tsx`.

**Build** (rename the existing `items` slice, per `docs/workshop.md §2`):
- `workers/api/db/schema/items.ts` → `presentations.ts` (fields: `id` ULID,
  `orgId`, `title`, `createdBy` = `c.var.userId`, timestamps). Export from
  `schema/index.ts`; `pnpm db:generate && pnpm db:migrate:local`.
- Copy `items-repo.ts` / `items-service.ts` / `items-controller.ts` →
  `presentations-*`; rename `item` → `presentation`.
- Rename the plan gate in `app/lib/plans.ts`: `maxItems` → `maxPresentations`
  (keep the free = 3 gate live so billing feels real in Phase 6); keep
  `usageRepo.increment` metering on create.
- Register the service in `workers/api/services/index.ts` and the controller in
  `workers/api/index.ts` (authed group).
- Copy `app/routes/app/items-list.tsx` → `presentations-list.tsx`; wire it as the
  `app` index route in `app/routes.ts`.
- Copy + rename the tests in `tests/`.

**Done when:** sign in → create a presentation → see it in the list; free-plan
limit blocks the 4th; `pnpm test` green; deploys.

### [x] Phase 2 — Slides (nested under a deck)

**Goal:** within a presentation, a user can add, reorder, and delete slides; each
slide stores its Konva scene as JSON. Managed as a list (visual editor is
Phase 3).

**Slice:** schema `slides` → `slides-repo` → `slides-service` →
`slides-controller` (nested under a presentation) → route
`app/routes/app/presentation-detail.tsx`.

**Build:**
- `workers/api/db/schema/slides.ts`: `id` ULID, `orgId`, `presentationId` (FK),
  `order` (integer), `scene` (JSON text column — the Konva stage JSON),
  timestamps. `pnpm db:generate && pnpm db:migrate:local`.
- `slides-repo.ts` — **scoped by `org_id` AND `presentationId`**; queries for
  list-by-presentation, insert, reorder, delete.
- `slides-service.ts` — verifies the parent presentation belongs to the org
  before any slide op (throw `NotFoundError` otherwise); assigns `order`.
- `slides-controller.ts` — routes under `/presentations/:presentationId/slides`;
  zod-validate the scene payload at the edge.
- Route: a presentation-detail page listing slides with add / reorder / delete.
- Tests for repo (real D1), service (mocked repo), controller (`app.request`).

**Done when:** open a deck → add/reorder/delete slides and they persist;
`pnpm test` green; deploys.

### [x] Phase 3 — Konva canvas editor

**Goal:** the core editor loop — open a slide, edit shapes/text/images on a
Konva.js canvas, and save the scene JSON back to the slide.

**Slice:** reuses the Phase 2 slides layers; adds a dedicated
`PATCH /presentations/:presentationId/slides/:id/scene` endpoint
(controller → service → repo) + the editor route
`app/routes/app/slide-editor.tsx`.

**Build:**
- `pnpm add konva react-konva`.
- Editor route: loads the slide's `scene` via `api-client.server`, mounts a
  Konva `Stage`/`Layer`, and saves on change (debounced) through the scene
  action. Keep the toolbar/shape logic client-side; the server only persists JSON.
- Add a thin `updateScene` path in the slides service/repo + zod schema for the
  scene shape; test it.
- shadcn UI for the toolbar/panels (`~/components/ui`).

**Done when:** edit a slide → refresh → your changes are still there;
`pnpm test` green; deploys.

### [ ] Phase 4 — Templates (standard, seeded)

**Goal:** ship with standard templates and let a user create a new deck **from a
template** (cloning its slides) as well as blank. Completes the first journey.

**Slice:** schema `templates` (+ `template_slides`) → `templates-repo` →
extend `presentations-service` with `createFromTemplate` → controller endpoint →
template picker in the "new presentation" UI.

**Build:**
- `templates.ts` schema — system templates are **global** (not org-scoped; e.g.
  `orgId` nullable or a reserved system org) seeded via a migration or a seed
  script; each template carries ordered slide scenes.
- `templates-repo.ts` — read-only list/get of system templates.
- `presentations-service.createFromTemplate(templateId)` — transactionally
  creates a presentation + clones the template's slides into org-scoped `slides`
  rows (respects the plan gate + usage metering).
- Controller: `POST /presentations` accepts an optional `templateId`.
- UI: template gallery on deck creation (blank tile + template thumbnails).
- Tests for the clone path and the read-only templates repo.

**Done when:** create a deck from a template → it opens with the template's
slides ready to edit; blank still works; `pnpm test` green; deploys.

### [ ] Phase 5 — Image uploads (r2-uploads) + embed into slides

**Goal:** a user can upload their own images and place them onto a slide. Both
uploads and AI generation (Phase 6) feed the same mechanic: an image URL becomes
a Konva `Image` node in the slide's `scene` JSON.

**Slice:** `r2-uploads` skill (R2 bucket + presigned/direct upload) → `assets`
schema → `assets-repo` → `assets-service` → controller → editor "Upload image"
panel.

**Build:**
- `pnpm install-skill r2-uploads`; follow its `SKILL.md` — uncomment the
  `r2_buckets` binding in `wrangler.jsonc` (`present-uploads`), create the bucket
  (`npx wrangler r2 bucket create present-uploads`), add `R2_PUBLIC_BASE_URL` to
  `.dev.vars` **and** `workers/env.d.ts`, then `pnpm cf-typegen`.
- `assets` schema — `orgId`, `createdBy`, `key` (R2 object key), `url`,
  `contentType`, `bytes`, timestamps. Scoped by `org_id`.
- `assets-service` — validates content-type/size, stores to R2 via the skill's
  adapter, records the row. Optionally gate count/bytes on the plan.
- Controller: upload endpoint (zod-validate metadata at the edge).
- Editor UI: an "Upload image" control that adds the returned URL as a Konva
  image node and saves the scene (reuses the Phase 3 save path).
- Tests: service with a mocked R2 adapter; controller with mocked service.

**Done when:** upload an image → it appears in the asset panel → drop it on a
slide → it persists in the scene; `pnpm test` green; deploys.

### [ ] Phase 6 — AI image generation (nano banana via fal)

**Goal:** generate an image from a text prompt with nano banana (Gemini image)
through the fal API and place it onto a slide.

**Slice:** schema `image_generations` → `image-generations-repo` →
`image-generations-service` (+ a `fal` adapter) → controller → editor UI
"Generate image" panel.

**Build:**
- Add `FAL_KEY` to `.dev.vars` **and** `workers/env.d.ts` (so `tsc` passes on a
  fresh clone); production via `wrangler secret put FAL_KEY`.
- `fal-adapter.ts` — plain `fetch` to the fal REST API (no SDK — mirror the
  `email-resend` adapter pattern), targeting the nano-banana / Gemini image
  model; takes the prompt, returns image URL(s).
- `image_generations` schema — `orgId`, `slideId?`, `prompt`, `status`,
  `resultUrl`, timestamps. Meter each generation with `usageRepo.increment`
  against `apiCallsPerMonth` and gate on the plan.
- Service kicks off the fal call and records the row; run the network call /
  persistence follow-up in `waitUntil(...)` so the response isn't blocked.
- Controller: `POST /image-generations` (zod-validated prompt); the editor polls
  or streams status, then drops the returned image onto the Konva canvas.
- **Storage note:** fal result URLs can be ephemeral — for durable images,
  reuse the Phase 5 R2 bucket: copy the result into R2 and store the R2 URL in
  `resultUrl`. Decide MVP (store fal URL) vs. durable (R2) at build time.
- Tests: service with a mocked fal adapter; controller with mocked service.

**Done when:** type a prompt in the editor → an AI image appears and can be
placed on a slide; usage is metered; `pnpm test` green; deploys.
